## 0. Ground truth about THIS project (read first)

This section restates the constraints so the rest of the document can be read years from now without the repo open. Everything here was verified against repo files on 2026-07-13.

### 0.1 The platform and scoring rule

- play.sportspredict.com "JTC Probability Cup": ~15 binary questions per WC2026 match, one probability submitted per question, scored in **RBP (Relative Brier Points)** — positive when your probability beats the crowd consensus on that question, negative otherwise. The exact RBP formula has **not** been reverse-engineered; it is confirmed non-linear in the Brier gap (`MODEL_ONLY_VS_MARKET_BLENDED_COMPARISON.md`, note on the RBP column). A regression-through-origin fit gave a rough scale of ~101 RBP per unit Brier improvement (`META_MODEL_LAB_NOTES.md` §4.4) but per-question scaling varies.
- The opponent is the **crowd**, not a sharp market. The crowd compresses toward 50% predictably; the project's crowd-bias regression is `crowd ≈ 0.514 + 0.61×(estimate − 0.5)`. This is the single most important structural fact: the task is *relative* forecasting against a soft benchmark, not absolute calibration.
- Track record: ~+2100 cumulative RBP over ~86 matches; 58–72% of questions beat the crowd depending on the sample window. Crowd's raw Brier (0.216–0.218) is *slightly better* than ours (0.218–0.219) even while we beat them on most questions — the edge concentrates in higher-leverage questions (`datasets/MASTER_DATASET_DICTIONARY.md`, `crowd_brier_score` note).
- The existing pipeline is classical statistics: point-in-time Elo → Poisson goals GLM (with Dixon-Coles low-score correction and negative-binomial dispersion) → ordered logit for W/D/L, plus ~18 hand-codified pricing rules (RULE1–RULE18) learned from postmortems, plus Smarkets sharp-market anchors when liquid.

### 0.2 The four prior ML rejections — what exactly failed and why

These are the load-bearing precedents. Any new ML proposal must explain what is different from each.

1. **Platt scaling diagnostic (2026-06-22, n=246)** — `ML_EXPERIMENTS_NOTEBOOK.md` Experiment 1. Fit `logit(P) = a + b·logit(our_estimate)`. Global b=0.51 (looks overconfident) but bootstrap 95% CI [0.12, 1.29] includes 1.0. Walk-forward b drifted to 0.196 — parameter instability, the hallmark of small-n overfitting. Applying the correction **worsened** test Brier by 0.028. Decision: do not apply; monitor at n≈350. Real structural finding that survived: HIGH-confidence (70%+) estimates underperform (+0.65 mean RBP vs +3.0 elsewhere) — consistent with favourite-longshot bias.
2. **Meta-model over rules and context (2026-07-05, n=404 usable rows / 42 matches)** — `META_MODEL_LAB_NOTES.md`. L2 logistic (C=0.3) and shallow HGB (depth 2, 60 iters) on 19 features, validated both walk-forward and GroupKFold-by-match. **Both models, under both schemes, lost to the zero-parameter `crowd_alone` baseline** (logreg by −0.039 to −0.043 Brier; HGB by −0.007 to −0.018). Key secondary finding: all rule-fired flags contributed ~nothing beyond continuous context features. Diagnosis in the doc itself: 42 matches is the true n (questions within a match share context), not enough for 19 features.
3. **RBP linear regression (2026-07-08, n=771 rows / 71 matches)** — `STATSBOMB_INTEGRATION_AND_STATS_TESTS_2026-07-08.md` §4. OLS with match-clustered SEs; in-sample R²=0.053; out-of-sample: walk-forward trivially better than predict-the-mean (0.2%), GroupKFold **worse** than the dumb baseline. Only robust finding: `is_player_prop` +4.87 RBP (player-prop suppression is the campaign's most reliable edge — cross-validates `WINNING_PATTERNS_SYNTHESIS.md`).
4. **Gap t-test / extremizing check (2026-07-08)** — same doc §6. Match-level (correct unit) Welch t-test: HIGH deviation from crowd → *lower* mean RBP (p=0.056, d=−0.46), opposite sign from the naive question-level test. Large deviations are higher-variance, not higher-reward. This kills naive "learn to extremize more" designs.

Common failure signature across all four: **small real in-sample signal, zero or negative out-of-sample signal, parameter instability across time splits, and the independent-unit count (matches, not questions) being ~10× smaller than the row count.**

Also on record: the **Ronaldo n=9 regression demo** (same doc §5) — in-sample R²=0.43, F-test p=0.68, LOO-CV loses to predicting the career mean. Kept as the project's canonical small-n overfitting exhibit.

### 0.3 The data reality

- `datasets/master_question_dataset.csv`: one row per (match, question); ~730–943 rows depending on backfill state, 66–85 matches, 107 columns. Grows ~15 rows/match, **2–3 matches left** in the tournament (SF ×2, final, maybe 3rd place). The dataset can never grow meaningfully before the tournament ends.
- Every question in a match shares match-level context → **the independent unit is the match (~70–85), not the question (~730+)**. Peduzzi events-per-variable logic caps an honest feature budget at ~8–12 features (`TRAINING_DATASET_STRATEGY_2026-07-07.md` §3).
- Known instrument bugs that any ML build must respect: (a) `elo_current_ratings.csv` is frozen pre-tournament — needs point-in-time replay per match (bug found in `MODEL_ONLY_VS_MARKET_BLENDED_COMPARISON.md` §0; replay scripts exist in `ml/backtests/`); (b) Q-numbers are not a join key across containers (project memory + dataset dictionary); (c) ESPN vs StatsBomb counting conventions differ systematically (fouls ×0.80, yellows ×0.73, SOT ×1.38, corners ×1.30 — `STATSBOMB_INTEGRATION_...` §3) so cross-source pooling requires a source fixed-effect or ratio correction.
- The **domain dataset** (historical team-match / player-match base rates) is the growable one: StatsBomb open data WC2018+2022 already flattened (256 team-match rows, 6,130 player-match rows, validated against known facts); extendable to Euro 2020/2024, Copa 2024, AFCON 2023 → ~314 matches / ~628 team-match rows / ~6,900 player-match rows at identical schema. This is where ML-ish methods have room; the meta/calibration dataset is where they have repeatedly died.

## A.1 Ch 01 — The Process Is Your Edge (`01_process_is_edge/README.md`)

**What the book says.** Durable performance comes from a disciplined research process, not model sophistication; ML for trading is an *adaptation* problem, not a model-selection contest. The workflow runs: data infrastructure → strategy research → **an evidence boundary separating tuning from evaluation** → deployment/monitoring, with a feedback loop that retrains, pauses, or retires a strategy as its edge decays (top-level `README.md`, "What's New"). The evidence boundary is maintained through three mechanisms (ch01 README, Learning Objectives): **trial logging** (count what you tried), **sealed holdouts** (data you never touch until the end), and **selection-aware evaluation** (corrections that account for the search). Section 1.4 insists regimes are "primarily a risk lens rather than a reliable timing signal." Section 1.5 is directly about the solo-practitioner case: institutions get governance for free through friction and review; independents must manufacture it via documentation, checkpoints, and explicit stop criteria. References include Harvey/Liu/Zhu (2016) factor-zoo, López de Prado "10 Reasons Most ML Funds Fail" (2018), Arnott et al. "A Backtesting Protocol in the Era of Machine Learning" (2018). Notebooks: `factor_regimes.py`, `macro_regimes.py` (GMM regime detection on AQR factor data / FRED macro) — least transferable part of the chapter for JTC.

**Transfer to JTC.** This chapter is essentially a formalization of what the project already does instinctively (ML_EXPERIMENTS_NOTEBOOK as trial log, the "7 Systematic" workshop notes, the rules graveyard). Three concrete upgrades it suggests: (1) **the remaining 2–3 matches are the sealed holdout** — the project's own note ("remaining WC2026 matches are the real out-of-sample test — don't contaminate it") is exactly the book's evidence boundary, and it implies any new ML must be specified *before* the FRA-ESP close, with the match's outcome scored once, not iterated on; (2) **selection-aware evaluation applied to the 18 rules**: each RULE was born from a postmortem "look" — the book would say the searched set is ≥18 (really more, counting revisions), so any per-rule performance claim needs a multiple-testing haircut (see A.3); (3) **retire/pause logic**: the book's decay loop maps to tracking per-rule RBP over time (which the project already resolved to do). Not transferable: regime detection on factor returns — with 2–3 matches left there is no "regime" to detect; the analogue (match-script regimes: blowout vs competitive) is already hand-modeled by RULE7/RULE15 and their revisions.

**Why this matters for JTC.** The four prior ML failures were caught precisely *because* the project already runs the ch01 process (honest out-of-sample tests, negative results documented). The book's message cuts the other way too: a fifth attempt that skips the process — e.g., tuned on all 730 rows and shipped to FRA-ESP — would be strictly worse than the first four, because the crowd-beating track record makes it tempting to trust an in-sample fit "just this once."

## A.2 Ch 06 — Strategy Research Framework (`06_strategy_definition/README.md`, `02_cv_foundations.py`, `exploration.md`)

**What the book says.** A strategy is "an executable decision process that has to be defined at decision time and evaluated as if it were live" (README §6.1). Before modeling, fix the invariants: universe, decision schedule, admissible information, score-to-position mapping, costs (§6.3). Distinguish **parameter tuning** (still the same strategy) from **mechanics changes** (a new strategy — new trial to count). §6.4 separates three metric layers that must not be conflated: *model diagnostics* (log loss, accuracy), *signal diagnostics* (IC), and *strategy outcomes* (Sharpe/P&L) — "do not use one metric to answer incompatible questions." §6.6: a narrow **baseline checkpoint** is a governance tool — earn the right to broaden the search only after the simple version passes timing/coverage sanity checks. §6.7: trial taxonomy (strategy → trial family → trial → run) with automatic run logging so search stays countable. The `exploration.md` file documents the chapter's own architecture: frozen config dataclasses, data contracts, and **strategy term sheets** — a structured document per strategy with a falsifiable hypothesis, implementation blueprint, feasibility gate, and validation plan (`StrategyTermSheet` dataclass with `hypothesis: FalsifiableHypothesis`, `feasibility: FeasibilityGate`, `validation: ValidationPlan`).

**`02_cv_foundations.py` (deep-read, 982 lines)** — the book's canonical CV treatment:
- **Decision-time admissibility** is the single design constraint from which everything follows: "At decision time t, which labeled samples were actually available?" Five named leakage channels (§1 of the notebook): label leakage (forward-looking labels overlap validation), standardization leakage (z-scoring with full-sample stats), threshold leakage (percentile labels from the full sample), survivorship leakage, point-in-time leakage (data reported after decision time).
- Three dataset roles (train / validation / sealed holdout); holdout touched once.
- Random k-fold "destroys temporal structure"; walk-forward (expanding vs rolling) is the default; **purging** (label buffer: end training ≥ one label-horizon before validation) and **embargo** (feature buffer: needed only when training can follow validation in time, i.e., CPCV/k-fold — zero in pure walk-forward). Buffers counted in *trading days* via a market calendar, not calendar days (a naive 21-calendar-day purge leaves 7 trading days of leakage).
- **Nested walk-forward**: outer loop advances the test window, inner loop retunes; "If λ*₁ ≠ λ*₂, that signals hyperparameter instability — a red flag for production deployment."
- **CPCV** (N blocks choose k held out → C(N,k) splits → multiple complete backtest paths); path dispersion feeds the Probability of Backtest Overfitting (Bailey et al. 2014).

**Transfer to JTC.** (1) The project's GroupKFold-by-match + walk-forward double scheme (META_MODEL_LAB_NOTES §4.3) is already the correct instantiation; the book adds the *purging* nuance, which for JTC translates to: if any feature is computed from a window of matches (e.g., rolling team form over last N matches), a match in the validation fold must not appear inside a training row's rolling window. The current master-dataset features (Elo pre-match, squad value, rest days) are point-in-time by construction, but the planned StatsBomb rolling base rates are exactly the feature type that needs this audit. (2) **Platt walk-forward b instability (0.51 → 0.196) is precisely the book's nested-walk-forward red flag** — the book gives a name and a decision rule ("red flag for production deployment") to what the project observed empirically. (3) The three-metric-layer separation maps to: model diagnostics = Brier/log-loss of the raw probability; signal diagnostics = beat-crowd rate / calibration by bucket; strategy outcome = RBP. The project has already discovered these decouple (crowd wins raw Brier, we win RBP) — the book's advice is to *optimize* on the model layer and *decide* on the strategy layer, never to tune micro-decisions directly on RBP (which is partially opaque and non-linear anyway). (4) The **term sheet** idea transfers almost verbatim: any fifth ML attempt should be written up *before fitting* — hypothesis, features (fixed budget), validation plan, promotion criterion, kill criterion. Synthesis: this document's §C.5 drafts that promotion rule.

**Why this matters for JTC.** The four failures all had good validation but no pre-registered *promotion criterion* — each experiment decided "do not deploy" after seeing results, which is fine for rejection but leaves ambiguity about what result *would* have justified deployment. Ch06's contribution is to force that number in advance.

## A.3 Ch 07 — Defining the Learning Task (`07_defining_the_learning_task/`)

**What the book says (README + notebooks).** Feature research starts before modeling: split-aware preprocessing (`SplitAwarePreprocessor`, NB02), execution-consistent labels (NB03), univariate feature-label triage (NB05/06), multiple-testing control (NB07), and causal plausibility checks (NB08).

**Labels (NB03 `03_label_methods.py`).** Fixed-horizon returns, time-series/cross-sectional percentile labels (thresholds computed *rolling*, never full-sample), **triple-barrier labels** (profit target / stop / max-holding, whichever is hit first), trend-scanning, plus **sample weights from uniqueness** and sequential bootstrap for overlapping labels, and an **effective sample size** computation (§6.6 of the notebook). §7.5 covers **meta-labeling**: "A primary model generates directional signals; triple-barrier labels determine whether each signal was profitable; a secondary (meta) model learns *when to act* and *how much to bet*... Meta-labeling turns a classification problem (direction) into a probability calibration problem (confidence)." Notably the book itself does **not** adopt meta-labeling in its case studies ("each case study trains a single model on one label horizon").

**Univariate evaluation (NB05 `05_signal_evaluation.py`).** Screening order: correctness at decision time → association with label (IC = rank correlation of feature with forward outcome) → shape (quantile monotonicity) → feasibility (turnover/costs). Includes a **binary label evaluation** section (§9) and **fold-aware evaluation** (§7): compute the metric per walk-forward fold and look at stability across folds, not the pooled number — "Full-Sample vs Fold-Level IC" can disagree. §7.1: within-time **permutation test** — shuffle the feature within each time slice to get a null that respects cross-sectional structure.

**Inference (NB06 `06_ic_inference.py`).** IC series are autocorrelated → naive t-stats overstate significance → use HAC (Newey-West) adjustment and **block bootstrap**; compute **effective sample size**; distinguish practical from statistical significance; **track-record planning** (§5): how many periods are needed before an IC of a given size is distinguishable from zero.

**Multiple testing (NB07 `07_multiple_testing.py` — deep-read, full 1,315 lines).** The quantitative core:
- Selection bias: testing N noise factors and picking the best gives E[max IC] ≈ √(2 ln N)·σ_IC. With 100 pure-noise factors, the "best" looks clearly positive.
- Pipeline: HAC p-values → Benjamini-Hochberg FDR (controls E[FP/discoveries]) or Holm-Bonferroni (controls P(any FP) — use when one false positive is unacceptable, "e.g., deploying a new strategy that incurs real capital risk").
- **Rademacher complexity / RAS (Rademacher Anti-Serum, Paleologo)**: when candidates are correlated, the effective hypothesis count is lower than the nominal count; the RAS penalty is an *absolute* deduction ≈ 2·R̂ on the IC, not a proportional haircut. In the notebook's demo the penalty (0.31) dwarfs the best noise IC (0.02). "Testing 100 variants of the same factor is less risky than testing 100 truly independent factors."
- Harvey/Liu/Zhu (2016): with 300+ published factors, demand t > 3.0, not 2.0.
- **Two-pass exploration/confirmation** (§5.1): screen everything on the first 80% (BH at α=0.10, promote on *fold stability* not peak performance), re-test only the promoted few on the held-out 20% (BH at α=0.05 over the reduced set). Smaller confirmation search set → milder correction → real power, while held-out data prevents double-dipping.
- **Deflated Sharpe Ratio** (§7, Bailey & López de Prado 2014): P(the best Sharpe among N tried is genuinely > 0), by comparing the observed best to E[max Sharpe under the null] given N trials, non-normality, and track length. **PBO** (§8): fraction of CPCV combinations where the in-sample-best config lands in the worst OOS half; PBO > 0.5 = severe overfitting. **MinTRL** (§9): minimum track record length for a Sharpe to reach significance — Sharpe 1.0 needs ~2.7 years of daily data; grows with N trials.
- Sobering applied result (§5.2): 13 real momentum/reversal/vol features on 14 years of ETF data — **zero survive BH-FDR at α=0.05** after HAC inference.

**Causal checks (NB08 `08_causal_sanity_checks.py`).** For features that survive statistical screening: timing placebo (shift the feature to a time it couldn't work; if the "signal" survives, it's an artifact), shared-driver check (does a common cause explain both feature and outcome?), regime heterogeneity (does the effect exist only in one regime = fragile), collider-bias demo, and a plausibility scorecard. "It helps reject weak stories early," not prove causality.

**Transfer to JTC — the Brier/RBP analogues of the Sharpe machinery.** This chapter is the most transferable in the book, and the mapping is nearly mechanical:
- *IC* ↔ per-question edge (our Brier term − crowd Brier term), or beat-crowd indicator. *IC series over time* ↔ per-match RBP series. *HAC/clustering* ↔ match-level clustering (the project's cluster-robust SEs and match-level t-test are the same move; the t-test sign flip in STATSBOMB doc §6 is the book's autocorrelation lesson in cross-sectional form).
- *Deflated Sharpe analogue for Brier*: the honest question for any rule or model is not "did it beat the crowd?" but "given that ~18 rules plus revisions were tried, what is E[max cumulative RBP] under the null that all rules are noise?" A concrete recipe (synthesis, using book components): permutation/bootstrap the per-question RBP contributions of each rule under a null (shuffle rule-fired flags across matches, preserving per-match structure), take the max across the 18-rule family per resample, and require an observed rule to clear that E[max] before treating it as real. Equivalent in spirit to DSR; needs no closed form.
- *MinTRL analogue*: how many matches until a per-question Brier edge of size d is significant given match clustering? With effective n = matches (~70–85) and observed per-match RBP std (~10 per question, high dispersion per match), the project can compute this directly — worth doing once, because it will likely show that a *small recalibration effect* (the plausible size of any learned correction) needs several hundred matches, i.e., it is formally undetectable this tournament. That is the honest quantitative version of "not enough data yet."
- *Two-pass exploration/confirmation*: with 2–3 matches left, the confirmation set is 30–45 questions. The book's design says: promote at most 1–2 candidate changes from all history, pre-register them, and score them on the remaining matches — do not run 10 experiments against the semifinal.
- *Causal/timing placebo for rules*: e.g., RULE15 (SOT suppression for losing-team forwards) can be placebo-tested by applying it to matches where the underdog condition did NOT hold and checking that it *doesn't* help there. The ARG-AUT postmortem (Sabitzer) was effectively a failed placebo discovered live at a cost of −31.86 RBP.
- *Meta-labeling*: the direct analogue is: primary "model" = existing rules+Elo/Poisson pipeline produces the direction (deviate from crowd up or down); a meta-model learns *when the deviation is trustworthy* (bet size = how far from crowd to submit). This is architecturally the most attractive ML design for JTC because it doesn't replace domain pricing, only gates it — but note it is *exactly* what the 2026-07-05 meta-model attempted (features: gap, abs_gap, context; outcome), and it lost to crowd-alone. The failure wasn't the architecture, it was n. See §C.3 for what would have to be different.

**Why this matters for JTC.** The chapter provides the vocabulary to say precisely why the four rejections were correct (selection bias + insufficient MinTRL) and gives the two tools that are actually usable *now* at tiny n: pre-registered confirmation sets, and null-simulation (E[max] under noise) for rule bookkeeping — neither requires more data, only discipline.

## A.4 Ch 08 — Financial Feature Engineering (`08_financial_features/`)

**What the book says.** A three-step filter turns a narrative into a feature spec: **horizon alignment** (does the feature's timescale match the label's?), **driver hypothesis** (what economic mechanism?), **role separation** (is this a *signal* or a *state variable*?). Feature knobs: reference frame, representation, aggregation — and the README distinguishes "hypothesis-changing choices from noise-control choices." §8.4: slow-moving data is *more* dangerous than fast data (reporting lags, revisions, repeated values "easily create fake evidence") — point-in-time discipline is the central constraint. §8.6 controls search: **one-knob-at-a-time exploration, within-family deduplication, multiple-testing-aware triage**. NB05 (`05_feature_selection.py`): IC screen → correlation filter → clustering/dedup → BH-FDR → stability selection via bootstrap IC → ML importance as a *late* step → post-selection verification. NB06 (`06_robustness_sensitivity.py`): parameter sweeps should show a **broad near-optimal plateau** (a needle-sharp optimum = overfit), RAS correction for parameter snooping, regime-conditional IC, and signal × state interactions evaluated for *incremental* information. NB07: event studies.

**Transfer to JTC.** (1) Signal vs state: in JTC terms, `our_estimate`/market quotes are signals; Elo gap, draw prob, is_player_prop, match-script expectations are *state variables* that should gate/scale, not predict directly. The meta-model treated everything as flat features; the book's framing suggests fewer features used as explicit interactions (e.g., "trust personal-history suppression only when |Elo diff| > X" — which is literally what the revised rules already are). (2) One-knob-at-a-time + dedup: the master dataset has 107 columns, many near-duplicates (fifa_ranking_diff vs elo_diff vs squad_value_diff all proxy strength). The meta-model's coefficients confirmed this (crowd_estimate + fifa_ranking_diff dominated; everything else ≈ 0). The chapter's pipeline says: dedupe the strength proxies to ONE before any fit. (3) The plateau test transfers directly to rule thresholds: RULE7's "<15% win prob" cutoff should perform similarly at 12% or 18%; if it only works at exactly 15%, it's fit to noise. Cheap to check, high value. (4) Horizon alignment warns against pricing tonight's match with 2018-era base rates unshrunk — which the StatsBomb work already handles via empirical-Bayes shrinkage toward tournament means.

**Why this matters for JTC.** The honest feature budget is ~8–12 (Peduzzi + 70 independent matches). Ch08 is the discipline for choosing *which* 8–12, and its answer is: one strength proxy, one market/crowd anchor, one question-family encoding, a couple of state gates — not 19 flat columns.

## A.5 Ch 09 — Model-Based Features (`09_model_based_features/README.md`)

**What the book says.** Outputs of *fitted* models are features: forecasts, filtered states, residuals, conditional volatility, regime probabilities, uncertainty summaries. Two rules dominate: (1) **fit within training windows and use filtered (not smoothed/future-informed) outputs** — e.g., an HMM's filtered state at t uses data up to t only; refit cadence must align with the walk-forward protocol; (2) **uncertainty and regime outputs are conditioning features, not stand-alone signals** (Learning Objectives). §9.4 is the conceptually novel bit: *the model's uncertainty is itself informative* — posterior widths distinguish a strong estimate from a weakly identified one even when the point forecast is identical.

**Transfer to JTC.** The project already lives this pattern: Elo (a fitted online rating), Poisson λs, and ordered-logit probabilities are all model-based features feeding a downstream pricing decision. Two upgrades: (1) **the point-in-time Elo bug is exactly this chapter's failure mode** — `elo_current_ratings.csv` frozen pre-tournament is a "stale filtered state"; the replay scripts (`ml/backtests/*_point_in_time_elo_replay.py`) are the fix and must run for FRA-ESP before any model output is trusted (the match MANIFEST already flags this). (2) Uncertainty-as-feature: the empirical-Bayes shrinkage machinery already computes an implicit n per team/player — carry `n_matches_history` (or the posterior width of the shrunk rate) alongside every StatsBomb base rate, and let it gate how far from the market/crowd a submission may deviate. That is a principled, zero-training-data version of what the failed meta-model tried to learn: "when is our number trustworthy?" — answered by measurement (sample size behind the estimate) instead of by regression. The Bruno Fernandes loss (personal floor from n=4 breaking) vs the Yamal win (real rate, but market far away) are distinguished exactly by this: how much evidence is behind the personal-history number vs how far it sits from the market.

**Why this matters for JTC.** It reframes "ML for FRA-ESP" away from "train a classifier" toward "make the existing fitted models point-in-time-correct and expose their uncertainty" — feasible in a day, no new labeled data required.

## A.6 Ch 05 — Synthetic Data (`05_synthetic_data/README.md`)

**What the book says.** Research is "adaptive and path-limited": one realized history + repeated search inflates backtests, so synthetic histories widen robustness analysis. Order of preference: **classical baselines first** (bootstrap, GBM, jump-diffusion, GARCH, Heston) because they are "interpretable, sample-efficient, and easier to validate"; learned generators (TimeGAN, Tail-GAN, Sig-CWGAN, GT-GAN, Diffusion-TS, LLM/GReaT tabular) only when a specific inductive bias is needed. §5.7: validation matters more than architecture — the **fidelity–utility–privacy** framework, with **TSTR** (Train on Synthetic, Test on Real) as the task-based benchmark.

**Transfer to JTC.** Deep generative models are flatly out of scope at n≈730 (they need orders of magnitude more data and the project already rejected far simpler learners). But the *classical simulation* idea transfers well and is underused here: the project's own Poisson/Dixon-Coles machinery IS a generative model of a match. Concrete uses: (1) **simulate the FRA-ESP slate** — draw 100k matches from the fitted λs (with DC correction and NB dispersion), plus simple within-match event processes for cards/corners/SOT calibrated to StatsBomb rates, and read all 15 question probabilities off the same simulation so they are *mutually consistent* (P(3+ goals), P(BTTS), P(ET), P(goal before hydration break) all from one joint model — kills RULE14-style consistency errors by construction); (2) **bootstrap the campaign**: resample matches (not questions) from the 66–85 match history to get honest CIs on cumulative RBP and on any rule's contribution — the "path-limited" correction for a one-tournament track record; (3) TSTR analogue: any learned calibration map can be sanity-checked on simulated slates where the true probabilities are known — if Platt-style correction can't recover a *known* miscalibration at n=730 in simulation, it certainly can't in reality (this converts "not enough data" from a judgment into a measured power curve). Synthesis: (3) is the single most decisive experiment available before deploying any calibration layer.

**Why this matters for JTC.** Small-n's only honest antidote that doesn't require waiting is simulation from the trusted domain model — and the project already owns the domain model.

## B.1 Soccer: the statistical modeling lineage

**Maher (1982), "Modelling association football scores."** The founding model: goals scored by each team as independent Poisson variables with means from multiplicative attack × defense parameters (plus home advantage). Referenced as the foundation Dixon-Coles built on (https://academic.oup.com/jrsssc/article-abstract/46/2/265/6990546 and the regista docs https://torvaney.github.io/regista/reference/dixoncoles.html).

**Dixon & Coles (1997), "Modelling Association Football Scores and Inefficiencies in the Football Betting Market," JRSS-C 46(2):265–280** (https://rss.onlinelibrary.wiley.com/doi/abs/10.1111/1467-9876.00065). Poisson regression with two famous modifications: (1) a dependence correction τ for low-scoring outcomes (0-0, 1-0, 0-1, 1-1) where the independent-Poisson assumption visibly fails; (2) a time-decay weight ξ down-weighting old matches in the pseudo-likelihood so ratings track current form. Fitted to English league/cup data 1992–95; demonstrated a positive betting return against 1995–96 bookmaker odds. **This is the project's own model family** — the JTC pipeline's "Dixon-Coles low-score correction with negative-binomial dispersion" is a direct descendant.

**Karlis & Ntzoufras (2003), "Analysis of sports data by using bivariate Poisson models," JRSS-D** (https://www.researchgate.net/publication/227719079). Models the *joint* score with a bivariate Poisson (shared covariance component λ₃ capturing common factors like pace/conditions), plus diagonal inflation for draws. Karlis & Ntzoufras (2009) later modeled the goal *difference* directly with the Skellam distribution (referenced at same source). *Transfer:* the covariance component matters for exactly the question families JTC prices — BTTS, total goals, "both halves" — where treating the two teams' goals as independent understates draw-ish outcomes; the DC τ correction is the cheap version of the same fix.

**Baio & Blangiardo (2010), "Bayesian hierarchical model for the prediction of football results," Journal of Applied Statistics** (https://www.researchgate.net/publication/46527004). Two conditionally independent Poissons with dependence induced through hierarchical priors; explicit **shrinkage of team-strength estimates** toward league means, and (in the full version) a mixture prior over "top/mid/bottom" team groups to avoid over-shrinking extreme teams. *Transfer:* this is the formal statistical home of what the project already does by hand — the empirical-Bayes pooling (`lambda_shrunk = (n·team_mean + k·global_mean)/(n+k)`, k=5) in `ml/statsbomb_baserate_test.py` is exactly a one-level hierarchical model with a fixed prior weight. The natural upgrade named in STATSBOMB doc's own "next tool" note (hierarchical/mixed-effects models) is precisely Baio-Blangiardo applied to counting stats (fouls, corners, SOT) instead of goals, with the prior weight *estimated* rather than set at k=5. The over-shrinkage warning transfers too: France/Spain are extreme teams; a global prior will pull their rates down too hard unless team-quality groups (or Elo as a covariate) are in the hierarchy.

**Why B.1 matters for JTC.** The lineage validates the project's core architecture (Poisson + DC + shrinkage) as the field's consensus solution for *exactly* this data regime — small samples per team, count outcomes, need for calibrated probabilities. Nothing in the modern ML literature has displaced it at international-tournament sample sizes.

## B.2 Soccer: ratings and evaluation

**Constantinou & Fenton pi-ratings** — "Determining the level of ability of football teams by dynamic ratings based on the relative discrepancies in scores between adversaries," JQAS (2013) (http://www.constantinou.info/downloads/papers/pi-ratings.pdf). Dynamic ratings updated from score discrepancies with diminishing returns for blowout margins and separate home/away components; the search-result summary describes it as outperforming standard Elo on EPL forecasting and profitable against bookmakers over five seasons. Implemented in penaltyblog (http://pena.lt/y/2025/04/14/pi-ratings-the-smarter-way-to-rank-football-teams/).
**Constantinou & Fenton (2012), "Solving the problem of inadequate scoring rules for assessing probabilistic football forecast models," JQAS** (https://www.researchgate.net/publication/227378917): argues W/D/L is ordinal, so use the **Ranked Probability Score (RPS)** rather than Brier/log-loss for match-outcome models. **Wheatcroft, "Evaluating probabilistic forecasts of football matches: the case against the ranked probability score"** (https://arxiv.org/pdf/1908.08980) pushes back: RPS's ordinality premise is shakier than assumed and the ignorance (log) score discriminates better. *Transfer:* mostly a non-issue for JTC — the platform scores **binary** questions with an RBP (Brier-relative) rule, so Brier is the right internal metric and RPS applies only if the project ever evaluates its W/D/L sub-model in isolation (where RPS is the community convention, e.g., the 2017 Soccer Prediction Challenge used it — https://arxiv.org/pdf/1908.08980 discusses this context).
**Elo variants**: Hvattum & Arntzen's goal-based Elo for football and the FIFA SUM Elo formula **[not verified online this session — from prior knowledge]**; "On Elo based prediction models for the FIFA World Cup 2018" (https://arxiv.org/pdf/1806.01930) covers Elo-based WC forecasting and comparisons. The project's K=60 World Cup Elo with the published G/We logistic formula is squarely within this family.

**Why B.2 matters for JTC.** Two points. First, the evaluation-metric debate is a reminder that *the scoring rule defines the game*: JTC's RBP is relative-to-crowd, so model quality should always be measured as Brier *gap* vs crowd on the same questions, never absolute Brier (the project learned this empirically — crowd wins absolute Brier while losing RBP). Second, pi-ratings are the one ratings upgrade with community evidence of beating Elo at club level; but with 2–3 matches left, swapping rating systems now has no payoff — file for a future season.

## B.3 Soccer: the modern xG/ML era

**The Open International Soccer Database + 2017 Soccer Prediction Challenge.** Dubitzky, Lopes, Davis & Berrar built a 216k-match database (Machine Learning 2019; https://link.springer.com/article/10.1007/s10994-018-5726-0, already cited in TRAINING_DATASET_STRATEGY). The challenge winner: **Hubáček, Šourek & Železný, "Learning to predict soccer results from relational data with gradient boosted trees," Machine Learning 108:29–47 (2019)** (https://link.springer.com/article/10.1007/s10994-018-5704-6) — manually constructed features (pi-ratings, PageRank-style ratings) + GBT beat both relational-learning and deep approaches. A 2024 follow-up ("Evaluating Soccer Match Prediction Models: A Deep Learning Approach and Feature Optimization for Gradient-Boosted Trees," https://arxiv.org/pdf/2309.14807, published in Machine Learning) found deep models still do not consistently beat well-featured GBDTs. **The consistent lesson: rating/feature engineering > model class; GBT on good ratings is the ceiling in practice.**

**Kaggle "Football Match Probability Prediction" (2022, Octosport/Sportmonks data)**: ~150k matches 2019–21, 860 leagues; winners used LSTM sequence models over the last-10-match team histories (https://www.sportmonks.com/blogs/football-predictions-kaggle-competition/, 12th-place LSTM writeup https://www.kaggle.com/code/lonnieqin/football-prob-prediction-lstm-12th-solution). *Synthesis:* the one public counterexample to "GBT always wins," and it required 150k matches of sequence data — five orders of magnitude more than JTC's independent-match count. It reinforces, not weakens, the small-n conclusion.

**xG and event-data research.** StatsBomb's open data ships `shot.statsbomb_xg` on every shot (verified in the project's own StatsBomb audit; 100% coverage even on 1970 matches). xG models are GBT/logistic classifiers over shot features (location, body part, situation); the project consumes them rather than fitting them. **VAEP / socceraction** (ML-KULeuven): Decroos, Bransen, Van Haaren & Davis, "Actions Speak Louder than Goals: Valuing Player Actions in Soccer," KDD 2019 (https://arxiv.org/pdf/1802.07127; docs https://socceraction.readthedocs.io/en/latest/documentation/valuing_actions/vaep.html). Values every on-ball action by its change to P(score) − P(concede) within the next k=10 actions, trained with CatBoost on SPADL-converted event streams (repo: https://github.com/ML-KULeuven/socceraction). *Transfer:* not the player-ratings use — the *pattern*: convert heterogeneous event JSON to a canonical action language (SPADL), then aggregate; JTC's `build_statsbomb_panel.py` already does the tabular analogue. VAEP-style short-horizon probabilities could in principle price timing-window questions ("goal between hydration breaks") better than flat Poisson rates, but that is a next-season project, not a semifinal one.

**Why B.3 matters for JTC.** The entire competitive-ML soccer literature agrees with the project's TRAINING_DATASET_STRATEGY conclusions: GBT over engineered ratings wins *when you have 100k+ matches*; below that, ratings + Poisson + shrinkage are the game. JTC sits far below that threshold on the meta/calibration problem and modestly below it even on the domain problem (~314 usable homogeneous matches).

## B.4 FiveThirtyEight SPI methodology

"How Our Club Soccer Projections Work" (https://fivethirtyeight.com/features/how-our-club-soccer-projections-work/) and the WC2022 version (https://fivethirtyeight.com/features/how-our-2022-world-cup-predictions-work/). Per the methodology page: each team carries an **offensive rating** (expected goals scored vs an average team) and a **defensive rating** (expected conceded); match forecasts generate two Poisson distributions around projected scores and convert them to a full scoreline matrix for W/D/L. Post-match rating updates use a **composite performance score** — not just goals but adjusted goals, shot-based xG, and non-shot xG (from ~550k matches back to 1888 + Opta play-by-play since 2010). SPI ratings/forecast CSVs are public: https://github.com/fivethirtyeight/data/tree/master/soccer-spi (frozen since 538's shutdown of the project). *Transfer:* SPI = the project's Elo→Poisson pipeline with two upgrades worth copying conceptually: (1) update ratings on *performance* (xG-ish composite), not just result — a lucky 1-0 shouldn't move ratings like a dominant 1-0; StatsBomb xG makes a mini version feasible for the point-in-time replay; (2) separate off/def ratings let you price "Spain 5+ SOT" off Spain's attack vs France's defense specifically, where a single Elo number conflates the two. The frozen soccer-spi CSVs include international ratings that could serve as an external sanity anchor for FRA/ESP relative strength **(synthesis; data is 2023-vintage, treat as stale prior only)**.

## B.5 MLB: the sabermetric forecasting tradition

- **Pythagorean expectation** (Bill James, late 1970s): win% ≈ RS^x/(RS^x+RA^x), x≈1.83 empirically (https://en.wikipedia.org/wiki/Pythagorean_expectation); Miller gave it a theoretical basis via independent Weibull-distributed run distributions, exponent ≈1.8 (https://arxiv.org/pdf/math/0509698). *Transfer:* the deep idea is **rate stats predict future wins better than win-loss records do** — process over outcome. JTC's analogue is using xG/SOT/shot-volume base rates rather than raw results to rate teams, which SPI (B.4) operationalizes.
- **log5** (James, 1981 Abstract): P(A beats B) = (pA − pA·pB)/(pA + pB − 2·pA·pB) — head-to-head probability from each side's overall rate (https://www.baseball-reference.com/bullpen/Pythagorean_Theorem_of_Baseball and Miller's head-to-head extension https://web.williams.edu/Mathematics/sjmiller/public_html/math/papers/PythagBothTeams10.pdf). *Transfer:* log5 is the odds-ratio composition rule — the same algebra JTC uses when combining a team's rate vs an opponent's suppression rate; it's the correct alternative to naive independence products (the QAT-SUI compound-question loss, RULE4, was a naive-product failure).
- **Projection systems** (guide: https://www.beyondtheboxscore.com/2016/2/22/11079186/projections-marcel-pecota-zips-steamer-explained-guide-math-is-fun): **Marcel** (Tango) — the deliberately minimal baseline: 3 seasons weighted 5/4/3, regress to league mean, apply an age adjustment; famously hard for complex systems to beat by much. **PECOTA** (Nate Silver, Baseball Prospectus, 2003) — similarity scores: forecast a player from the career paths of historically comparable players (https://www.mlb.com/glossary/projection-systems/player-empirical-comparison-and-optimization-test-algorithm). **ZiPS** (Szymborski) — recency-weighted baseline (4 years; 3 for very young/old) + historical comparables. **Steamer** (Cross et al., 2008) — weighted averages regressed toward league average with per-stat weights *set by regression on past players*, plus aging curves.
  *Transfer:* three exportable principles. (1) **The Marcel lesson**: a transparent weighted-average-plus-regression-to-mean baseline captures most achievable accuracy; every JTC player-prop estimate should first be "Marcel-ized" (recency-weighted personal rate, shrunk to position/tournament mean) before any cleverness. This is what the winning player-prop suppressions (Pulisic, De Bruyne, Yamal) implicitly did, and what the Bruno Fernandes loss skipped (no shrinkage of an n=4 floor). (2) **Similarity-based projection** (PECOTA/CARMELO) is the field's answer to tiny per-player samples — borrow strength from comparable players. At JTC scale: price "Mbappé scores" partly from the StatsBomb rate of *comparable elite forwards vs elite defenses*, not just Mbappé's own 9-ish WC matches. (3) Every serious system regresses to the mean by an amount tuned per statistic — cards/fouls regress differently from SOT; the k in JTC's empirical-Bayes pooling should differ per stat family (estimable from the 256-row StatsBomb panel via variance decomposition).
- **Statcast era**: xwOBA prices each batted ball from exit velocity + launch angle (+ sprint speed) against historical comparables (https://www.mlb.com/glossary/statcast/expected-woba). It is formally *descriptive*, but the actual-minus-expected gap identifies luck and predicts regression (https://www.baseballprospectus.com/news/article/40026/prospectus-feature-siren-song-statcasts-expected-metrics/ — with the caveat that for pitchers the "expected" metrics can mislead). *Transfer:* identical logic to xG-vs-goals in soccer: when a team's goals exceed its xG, expect regression — directly usable in the FRA/ESP point-in-time form read.
- **Calibration of baseball win-probability models**: I did not find a single canonical paper this session; the topic is scattered across Baseball Prospectus / academic articles (e.g., Bayesian in-game WP for NBA at https://arxiv.org/pdf/2207.05114 is the nearest verified analogue). **Flagged as not properly sourced.**

## B.7 Forecasting communities: GJP, Metaculus, prediction markets

**Good Judgment Project.** "The Good Judgment Project: A Large Scale Test of Different Methods of Combining Expert Predictions" (Ungar, Mellers, Satopää, Baron, Tetlock et al., https://www.researchgate.net/publication/286280462). Key findings per the AI Impacts synthesis (https://aiimpacts.org/evidence-on-good-forecasting-practices-from-the-good-judgment-project/): aggregation + **extremizing** (pushing the pooled probability away from 0.5, e.g. a weighted mean of 0.7 → 0.9) won the IARPA tournament years; Satopää et al.'s partial-information framework formalizes when/how much to extremize (each forecaster sees a subset of information, so the average is under-confident relative to the union of information). Two critical caveats recorded by the same source: extremizing "successes... were a fluke"-adjacent (expected small gains most of the time, big losses occasionally), and **the aggregation algorithm helps ordinary forecasters most — superforecasters were good enough that it didn't help much**. Mellers et al. 2024 update: https://journals.sagepub.com/doi/10.1177/17456916231185339.
*Transfer — this is the single most JTC-relevant community finding.* JTC's whole edge is structurally "be the extremizer against an under-extremized crowd": the crowd regression (crowd ≈ 0.514 + 0.61·(estimate−0.5)) says the crowd compresses ~40% toward 50%, so a well-calibrated forecaster automatically beats it by being appropriately extreme. But GJP's caveat is exactly the project's own match-level t-test finding (§0.2 #4): extremizing is small-gains-usually / large-losses-sometimes, and beyond-your-true-belief extremizing is negative-EV under a proper scoring rule (STRATEGIC_MARGIN_PUSH already proved this internally). Synthesis: the correct posture is *calibrated conviction, not strategic exaggeration* — deviate exactly as far as the evidence supports, no further; the anti-correlation of |gap| with match RBP shows the project has at times crossed that line.

**Metaculus.** Community prediction = recency-weighted **median** of latest forecast per user (https://www.metaculus.com/faq/); scoring = log-score-based Baseline and Peer scores (https://www.metaculus.com/help/scores-faq/); public track record shows tight calibration with slight overconfidence in the 85–95% band (https://oddsreference.com/predictions/platforms/metaculus). *Transfer:* (1) medians resist the extremist-outlier problem that means have — when JTC aggregates several internal signals (model, market, personal history), a trimmed/median combination is more robust than averaging; RULE1's avg(Smarkets, crowd) is a two-source version. (2) Metaculus's slight overconfidence at 85–95% matches JTC's own HIGH-bucket (70%+) underperformance — the same bias shows up in the best-calibrated communities, supporting a standing cap/shade at the top of the range (RULE6's 0.90–0.95 cap is right on the evidence).

**Prediction-market efficiency.** Favourite-longshot bias documented across betting markets; **Snowberg & Wolfers (2010), "Explaining the Favorite-Long Shot Bias: Is it Risk-Love or Misperceptions?", JPE** — data support *misperceptions* over risk-love (https://www.nber.org/system/files/working_papers/w15923/w15923.pdf); surveys: Ottaviani & Sørensen (2008) (via same search). General finding: liquid sports markets are near-efficient; odds-based strategies don't yield significant long-term profits (https://www.researchgate.net/publication/351985837). Kaunitz et al. (2017), "Beating the bookies with their own numbers" (https://arxiv.org/abs/1710.02824) — already the project's strategic frame (TRAINING_DATASET_STRATEGY §1). *Transfer:* the JTC crowd is NOT a money-staked market — it is a Metaculus-like crowd *without* recency weighting or scoring feedback pressure, hence lazier: compression toward 50%, favourite-longshot bias on player props (market mid 41% for a De Bruyne S/A vs true ~24%). The efficient-market literature says don't fight Smarkets on match odds (RULE1 already complies); the inefficiency literature says harvest the crowd's compression and its prop-family biases — which is the campaign's realized P&L pattern.

## B.8 Small-sample probabilistic evaluation

- **Murphy decomposition**: Brier = uncertainty − resolution + reliability (Murphy 1973; overview https://www.emergentmind.com/topics/brier-score; generalization: "Simplifying and generalising Murphy's Brier score decomposition," QJRMS, https://rmets.onlinelibrary.wiley.com/doi/abs/10.1002/qj.2985; two extra binning-artifact components: Stephenson et al., Weather & Forecasting 2008, https://journals.ametsoc.org/view/journals/wefo/23/4/2007waf2006116_1.xml). *Transfer:* the decomposition names JTC's situation precisely — vs the crowd, JTC wins on **resolution** (sorting questions away from base rates in the right direction) while roughly tying/losing on **reliability** (calibration). Any learned layer targets reliability, which is the *smaller* component of the edge; that asymmetry is another reason the calibration-layer experiments kept failing to add value.
- **Sampling uncertainty of Brier/BSS at small n**: Bradley, Schwartz & Hashino (2008) built CIs for Brier and Brier *skill* scores (referenced in the emergentmind overview above); Siegert, "Variance estimation for Brier Score decomposition" (https://arxiv.org/pdf/1303.6182). Reliability-diagram bins are unstable at small n (JTC's 11 buckets × 24 questions are near the minimum). *Transfer:* put a CI on everything; the project already does (bootstrap CI on Platt b), keep doing it for cumulative-RBP claims — resample **matches**, not questions.
- **Tournament/rank objectives change optimal strategy.** Kaggle March Machine Learning Mania: Lopez & Matthews (2014 winners) wrote on luck's role; Landgraf's 2017 winner interview describes deliberately "optimizing luck" — choosing predictions to maximize P(top rank), not expected score, including 100%-ing one bracket outcome across paired submissions (https://medium.com/kaggle-blog/march-machine-learning-mania-1st-place-winners-interview-andrew-landgraf-f18214efc659). *Transfer:* JTC is a rank tournament too (stuck ~#50). Landgraf's logic says a rank-maximizer should add variance late if behind. BUT the project's own data (BRA-MAR 1.00 disaster, −51.97; the |gap|-vs-RBP t-test) shows variance-adding via overconfident singles is how this campaign takes its worst losses, and RBP (unlike bracket scoring) punishes each overreach immediately. Synthesis: if a deliberate variance play is ever justified (final match, needing to jump ranks), do it via *many small correlated deviations on well-evidenced questions*, never via one extreme number — and decide this as bankroll strategy, not forecasting.

# Task C — Synthesis for THIS project

## C.1 What the three evidence streams agree on

Independently, the ML4T book (A), the sports literature (B), and the project's own four experiments (0.2) converge on five statements:

1. **Feature/rating engineering beats model class at every n; model class only matters at huge n.** ML4T case studies: GBT strongest tabular model, but only after the feature pipeline; Hubáček 2019 and the 2024 follow-up: engineered ratings + GBT with 200k matches; JTC's meta-model: crowd_estimate + one strength proxy carried all the signal, 17 other features and all rule flags added nothing.
2. **At small n, the winning technology is partial pooling / shrinkage, not discriminative learning.** Baio-Blangiardo, Marcel/Steamer regression-to-mean, PECOTA similarity, 538's ½-carryover, JTC's own empirical-Bayes StatsBomb test. Nothing in either literature beats a well-shrunk base rate at n<1000 independent units.
3. **Calibration layers need on the order of 10³ points and target the smaller component of JTC's edge (reliability, not resolution).** Niculescu-Mizil & Caruana small-n guidance; Murphy decomposition (B.8); Platt and meta-model failures at n=246–404.
4. **The evaluation protocol is non-negotiable and JTC's is already correct**: walk-forward + GroupKFold-by-match + cluster-aware inference + zero-parameter baselines. Every "signal" the project has found in-sample died under this protocol — which is the protocol working, not failing.
5. **The crowd is beatable by calibrated conviction, not by strategic exaggeration.** GJP extremizing caveats + the project's match-level |gap| t-test + the proper-scoring-rule argument all say: deviate exactly as far as evidence supports.

## C.2 Why the four prior attempts failed — a unified account

All four attacked the **meta/calibration problem**: learn a map from (our estimate, crowd, context) → better probability. That problem has three structural properties that doom it here:
- **The independent n is matches (42–85), not questions.** Every within-match question shares context; the effective sample is two orders of magnitude smaller than the row count suggests.
- **The baseline (crowd-alone / raw estimate) has zero parameters** and therefore zero overfitting risk. A challenger spends variance it cannot afford — at these n's the variance cost of even 2 fitted parameters (Platt) exceeds the bias gain.
- **The learnable effect is small by construction.** The project's raw estimates are already roughly calibrated (Platt CI includes identity); the residual miscalibration worth ~0.001–0.005 Brier sits under sampling noise of ~0.01+ at n≈400. A MinTRL-style calculation (A.3) would show detection needs several×10² more *matches*. Not a modeling failure — an information-theoretic wall.

What the failures did NOT show: that ML is useless on the **domain problem** (historical base rates → price a fresh granular question). That problem has n = 256–628 team-match rows (growable to ~6,900 player-match rows), a *real* outcome as target (not "beat a soft crowd"), effect sizes that are large (team foul rates differ by ×2), and it is where the campaign's proven edges (player-prop suppression, GD-calibration) already live. The StatsBomb base-rate test (+93 RBP-equivalent standalone, best-used-as-prior conclusion) is the existence proof.

## C.3 Candidate designs, ranked

Ranked by (expected value × feasibility before tournament end), with failure modes and what differs from the four precedents.

**1. Simulation-based consistent slate pricing (deploy now).** Use the existing fitted Poisson/DC/NB machinery, with point-in-time-replayed Elo, to simulate ~100k FRA-ESP matches (goal times via the within-match intensity profile; cards/corners/SOT as conditionally-Poisson processes with StatsBomb-calibrated rates given match state). Read every P off one joint simulation: 3+ goals, BTTS, ET, timing windows, SOT thresholds — mutually consistent by construction. *Why viable:* zero new labeled data; it is inference from the already-trusted domain model, not learning. *Failure modes:* mis-specified within-match dynamics (e.g., card intensity rising when trailing — the ARG-AUT lesson) will bias correlated questions together; mitigate by calibrating state-dependent intensities on StatsBomb event timestamps and sanity-checking marginals against Smarkets. *Different from precedents:* nothing is fitted to the 730-row meta dataset at all.

**2. Hierarchical partial-pooling models per stat family on the domain panels (build during/after tournament).** For each family (fouls, cards, corners, SOT, offsides): a Poisson/negative-binomial GLMM with team random effects, opponent-strength covariate (Elo), tournament-era effect, and `data_source` fixed effect for the ESPN/StatsBomb convention gap. Fit with footBayes/Stan or lme4. This replaces the fixed k=5 shrinkage with estimated pooling per family and yields *uncertainty intervals* to gate deviations (A.5). *Why viable:* n=426 unified team-match rows now; the model class is the literature consensus for exactly this regime (B.1); targets real outcomes. *Failure modes:* (a) cross-source pooling without the fixed effect silently biases rates (measured: SOT ×1.38); (b) 2018-era rates go stale for churned squads — include era decay; (c) elite-team over-shrinkage (Baio-Blangiardo mixture warning) — France/Spain sit in the tail; (d) it prices the *question input*, not the submission — human/rules still own the final number, which is by design. *Different from precedents:* different dataset, different target, 10× effect sizes; the precedent it extends (StatsBomb base-rate test) *worked*.

**3. ML as feature-generator feeding the existing rules (conditional go).** GBT (LightGBM, shallow, heavily regularized) per market family on the player-match panel (6,900 rows) to produce prior rates for player props (P(1+ SOT | minutes-expected, position, opponent-defense-Elo, team λ)); output enters the pipeline as one more evidence source under RULE5-style blending, never auto-submitted. *Why viable:* the one place row counts approach GBT territory; player props are the campaign's proven edge; the human blend contains damage. *Failure modes:* leakage via features computed from the same match (audit per A.2 purging rules); minutes/lineup uncertainty dominates (Pulisic-full-90 shows lineup status is the real variable — the model must condition on expected minutes, which is itself a forecast); roster churn 2018→2026 (prefer role/position features to player identity). *Different from precedents:* trained on 6,900 domain rows against real outcomes, not 400 meta rows against a soft crowd.

**4. Regularized logistic per question-family calibration (research only, not for deployment this tournament).** One or two parameters per family (e.g., family-specific intercept shift on logit(our_estimate)), partial pooling across families (hierarchical prior on the shifts). The is_player_prop +4.87 RBP finding and the HIGH-bucket overconfidence pattern are the two family effects with any evidence. *Why borderline:* it's the Platt failure with better structure; per-family n is 50–200 questions ⇒ 15–40 effective matches. *Gate before any use:* the A.6 TSTR-style power check — inject a known family-level miscalibration into simulated slates at n=730 and verify the fitted layer recovers it out-of-sample; if it cannot recover a *known* effect in simulation, drop the idea. *Failure mode if skipped:* exactly Platt/meta-model redux.
 
**5. Calibration-layer-only learning (global Platt/beta/isotonic): rejected — do not revisit this tournament.** Tested twice, structurally information-starved (C.2). Earliest revisit: a future season with n≥1,500 questions AND per-family pooling, or platform history from other users if ever obtainable.

**6. GBT over pooled question features with match-grouped CV (the meta-model, again): rejected.** Already run under the correct protocol; lost to crowd-alone under both schemes. Nothing about the data will change materially in 2–3 matches. Re-running it post-tournament with the backfilled 12-match gap (§0.3) is legitimate bookkeeping, not a deployment path.

**7. Meta-labeling gate ("when to trust our deviation"): right architecture, unlearnable at current n — substitute the measured version.** Instead of learning the gate, compute it: deviation allowed ∝ evidence behind the number (posterior width from design #2, sample size behind a personal-history read, market liquidity). This encodes the Bruno-Fernandes-vs-Yamal distinction with zero fitted parameters. Revisit as a *learned* gate only when the meta dataset is several times larger.

**8. Deep learning of any kind (LSTM/transformer/TabPFN-style): rejected without experiment.** The one public soccer LSTM win used 150k matches (B.3); every small-n source says no. Running the experiment would only spend evidence-boundary budget.

## C.5 The decision rule for any future ML deployment here

Written to be enforceable years from now:
1. **State the target problem class first.** Meta/calibration (n = matches) vs domain (n = team/player-matches). Anything meta-class at n < ~300 matches is presumptively rejected; the burden of proof is a simulation power analysis (design #4 gate) showing the effect is recoverable at the available n.
2. **Pre-register before fitting:** hypothesis, feature list (≤12, deduplicated strength proxies), validation scheme (walk-forward AND GroupKFold-by-match, purged if any rolling features), baselines (crowd-alone, raw estimate, 50/50 blend, current pipeline), and the promotion criterion as a number — e.g., "beats the best baseline on both schemes by more than the bootstrap-resampled (by match) 90% noise band."
3. **Count the search.** Every variant tried is a trial; apply BH-FDR across the family or an E[max under null] simulation before believing the best one (A.3). Rules count too: any new RULE gets a placebo test (does it 'work' where its mechanism shouldn't apply?).
4. **Shadow-deploy first.** A promoted model prices the slate in parallel without being submitted for at least one full match; compare realized RBP-equivalent. Only then may it touch a submission, and only as one blended evidence source with a human override.
5. **Keep the graveyard.** Failed experiments stay documented with their n and the reason (this file + ML_EXPERIMENTS_NOTEBOOK). A future attempt must cite which graveyard entry it supersedes and what changed (more data / different problem class / different target), or it doesn't run.

**Bottom line.** For this tournament: ML the *pricing inputs* (point-in-time ratings, shrunk hierarchical base rates, simulated joint slates, Marcel-ized player rates) and keep *learning the submission map* off the table — the four graveyard entries all died there, and 2–3 matches cannot resurrect it. The credible ML build is the domain-side one, and most of its value (StatsBomb panels, shrinkage, replay scripts) already exists in this repo; the hierarchical upgrade (design #2) is the one genuinely new, genuinely viable piece of machine learning this project can still build.
